Neo UI Framework

A production-grade UI framework for Unity: addressable views, signal-driven wiring, a visual flow graph for navigation, a token-based theme system, and SDF vector graphics that batch on a single material. Everything can be authored two ways — by hand in the editor, or from a JSON spec that generates (and safely re-generates) the whole UI.

Extensible by designEvery fixed set — widget kinds, presets, theme tokens, flow nodes — has a registration seam. You never fork the package to add your own.
Addressed by nameViews, buttons, signals and streams are identified by Category/Name strings, not object references or GUIDs. Wiring survives refactors and is readable in plain JSON.
Spec ⇄ prefab round-tripA JSON spec is the source of truth; prefabs are its materialization. Generate, export, diff, and three-way-merge without losing hand edits.
Loud failures, quiet editorEvery string-addressed lookup that misses logs a warning. Editor tooling never polls, never scans with reflection, never animates chrome.

What's in the box

Containers

Views, panels, popups, tooltips, overlays, and responsive roots with show/hide transitions.

Widgets

Buttons, toggles, sliders, dropdowns, tabs, steppers, counters, badges — all id-addressed and signal-emitting.

Animation

Five-channel UIAnimation (move / rotate / scale / fade / color), preset library, pooled tween engine.

Flow graphs

A node-based state machine for screen navigation with triggers, history, and named view transitions.

Signals

Named event streams (Category/Name) decoupling senders from receivers, with optional typed payloads.

Theming

Color tokens × variants (Dark/Light/…), text styles, shape styles — switch a variant and the whole UI re-skins live.

NeoShape

SDF vector shapes (rounded rects, pills, rings, glyphs) with gradients, borders, shadows, effects and UI particles — one shared material.

Editor suite

Hub, Flow Graph editor, Design System window, Gallery, Palette, Command Palette, ID Database manager, setup wizards.

Agent Bridge

A JSON request/response protocol so tools and AI agents can generate, validate, preview, and sync UI with the editor open or headless.

Getting started

From an empty project to a themed, navigable UI in five steps.

1 · Open the Hub

Tools → Neo UI → Hub is the central launcher. Every window, wizard, and advanced action is organized there by category (Author / Data / Setup / Advanced), with search. If you only remember one menu item, remember this one — or press Ctrl+K for the Command Palette, which finds views, widgets, and windows by typing.

2 · Run project setup

Tools → Neo UI → Setup → New Project Setup… walks you through choosing a theme and installing the pieces you want. It creates the single settings asset (Resources/NeoUISettings), the starter kit prefabs, widget presets, animation library, fonts, effects, and transition library. Each piece can also be created or repaired individually from the same Setup menu, and Reset To Clean Slate… uninstalls anything setup created, per component.

3 · Learn from the showcases

Every user-visible feature has a demo scene under Assets/Showcases/ — buttons, toggles, tabs, popups, animations, effects, particles, transitions, a settings menu, and a complete game UI with flow navigation. Open one, press play, then read its spec in Assets/Showcases/Specs/ to see exactly how it was described. See Showcases.

4 · Build your first screen — pick a path

Native authoring: drag widgets from the Palette window (or use starter-kit prefabs), give each an id via the inspector's category/name picker, and wire navigation in the Flow Graph Editor.

Spec authoring: write a JSON spec (there's a JSON Schema at Assets/docs/neo-spec.schema.json for IDE autocomplete) and run Tools → Neo UI → Advanced → Generate From Spec…. The generator creates prefabs, registers ids, builds flow graphs, and writes a baseline for future safe re-generation. See The spec format.

5 · Validate

Tools → Neo UI → Advanced → Validate runs hard contract checks (theme coverage, flow references, dead ids) plus soft design lints (WCAG contrast, tap targets, spacing scale) and an interactivity lint that flags buttons and tabs that do nothing. Run it after every generate. See Validation.

Tip

The framework uses the New Input System exclusively. If pointer events aren't reaching your UI, check that your EventSystem uses the InputSystemUIInputModule, not the legacy StandaloneInputModule.

Core concepts

Category/Name ids

Almost everything is addressed by a two-part string id: a category and a name — Menu/Main, Shop/BuyButton, Audio/Muted. Typed id classes exist per widget family (ViewId, ButtonId, ToggleId, SliderId, DropdownId, PanelId, TagId, StreamId), each backed by an IdDatabase asset that powers searchable pickers in every inspector, with quick-add and jump-to buttons.

Ids also drive naming: a component implementing INeoIdOwner gets a canonical GameObject name derived by NeoIdNaming (Type-Category_Name, e.g. UIButton-Shop_Buy), shared between hand authoring and generation — so hierarchies stay predictable and greppable.

Signals as glue

Components rarely hold references to each other. Instead they publish and subscribe to signal streams — named channels like Audio/Muted. A button click, a toggle flip, a flow-graph transition, a particle burst: all can be expressed as "send this signal" / "react to this signal". This keeps prefabs self-contained and makes wiring visible as data.

Spec is source of truth; prefab is materialization

When you use the spec workflow, the JSON spec describes the UI and the generator materializes it into prefabs, id registrations, presets, and flow assets. A baseline file (.neo-baseline.json) records the spec the assets were last generated from, so the framework can always tell what you edited by hand versus what the spec says — and merge instead of clobbering. Hand-authoring without specs is fully supported too; the two paths produce identical widgets (verified by round-trip tests).

WYSIWYG baking

The state you see in the prefab is the state you get at runtime start: colors are baked, start values are serialized, disabled things start disabled. There is no "wait one frame for the theme to kick in" flash.

Invariant

No silent failures. Any string-addressed lookup that misses — showing an unknown view, starting an unknown flow, sending to a misspelled command — logs a warning naming what was looked up. If something "does nothing", the Console will say why.

Invariant

Lifecycle ordering. Registries populate in OnEnable; anything that commands another object defers to Start(), because enable order across objects is arbitrary. Follow the same rule in your own scripts to avoid enable-order races.

Containers & views

Containers are the things that show and hide. Everything visible lives inside one.

ComponentWhat it is
UIContainerBase visibility container: show/hide with transitions, animator registration, interruption handling, auto-hide and auto-select behaviors.
UIViewA screen. Addressed by Category/Name, controllable from anywhere via static API, navigated by flow graphs.
UIPanelTab-content container, kept in sync with a UITab's selection; optional fade-in on enable.
UIPopupModal/popup looked up by name from PopupDatabase, configured with a fluent API, queued FIFO (one visible per queue).
UITooltipTooltip with follow/anchor positioning and hover triggering; anchors resolve via named UITag targets.
UIOverlayMarks children as z-stacked (overlay layout) rather than layout-group stacked.
UIResponsiveRootResponsive layout root managing breakpoint-aware canvas scaling.
UIResponsiveGridColumnsGrid whose column count adapts to the active screen breakpoint.

Controlling views from code

// Show / hide by address — from anywhere, no references needed
UIView.Show("Menu", "Main");
UIView.Hide("Menu", "Settings", instant: true);
UIView.ShowCategory("HUD");      // every view in a category
UIView.HideAllViews();

// Popups: fetch by name, configure fluently, then show
UIPopup.Get("Confirm")
       .SetTexts("Delete save?", "This cannot be undone.")
       .SetEvents(OnConfirm, OnCancel)
       .SetQueue("dialogs")
       .Show();

Every show/hide accepts an instant flag and an optional UIAnimation override. Containers handle interruption gracefully — hiding mid-show reverses from the current pose rather than snapping.

Interactive widgets

All widgets carry a Category/Name id, emit signals, and animate their selection states through the same animator model.

Button behavior triggers

Each UIButton hosts a list of UIActionBehaviours — one per trigger — each firing a UnityEvent, an optional signal, and honoring an optional cooldown:

ClickDoubleClickLongClickRightClick PointerEnterPointerExitPointerDownPointerUp SelectedDeselectedSubmit

Selection states

Widgets built on UISelectable expose five states — Normal · Highlighted · Pressed · Selected · Disabled — consumed by pluggable animators (UISelectableColorAnimator, UISelectableUIAnimator) via SelectionStateRelay. Spec-generated buttons come in variants primary secondary danger light ghost and sizes sm md lg.

Invariant

Dead-interaction lint. A button or tab with no flow trigger, signal, popup, view command, or event is flagged by Validate — a control that looks alive but does nothing never ships silently.

Animation & tweens

One composite animation type, a preset library, and a pooled tween engine ticked by a single central service.

UIAnimation — five channels

MoveRotateScaleFadeColor

Each channel has its own TweenSettings: duration, delay, loops, play mode (Normal · PingPong · Spring · Shake), and easing — either one of ~30 built-in Ease curves (Sine/Quad/Cubic/Quart/Quint/Expo/Circ/Back/Elastic/Bounce in In/Out/InOut flavors, plus Linear and Spring) or a custom AnimationCurve. Move supports directional presets (UIMoveDirection: Left, Top, BottomRight, CustomPosition, …) so "slide in from left" is one dropdown.

The preset library

UIAnimationPreset assets store named, reusable animations, addressed — as always — by Category/Name. Any preset asset dropped under the library folder is auto-discovered; no registration needed. The setup wizard installs 40 presets across seven categories:

Applying presets

In the inspector: every animator slot (the "Show" animation on a container animator, the "Hover" animation on a selectable) has a picker grouped by category. Hovering a preset plays a live preview; clicking it copies the animation data into the slot. Presets are copy-on-apply, not linked — editing a preset later never changes widgets that already used it, and you can freely tweak the copy per-widget.

In a spec: reference presets by full name on the matching field:

{ "id": "Menu/Main",
  "showAnimation": "Show/FadeIn",
  "hideAnimation": "Hide/SlideOutLeft" }

In the Design System window: the Motion tab is the library's home — a searchable browser on the left, the selected preset's five-channel editor on the right, with play/scrub preview on a preview stage or any scene RectTransform.

Animator roles

NeoAnimatorRoles names the standard animation slots, and NeoUISettings maps each role to a default preset — so every new widget animates consistently with zero per-widget setup, and re-mapping a role restyles the whole project's motion at once:

View/ShowView/Hide Button/HoverButton/Press Selectable/NormalSelectable/SelectedSelectable/Disabled Toggle/OnToggle/Off LoopOneShot

Each role suggests matching preset categories first in the picker (View/Show surfaces the Show category, and so on).

Animator components

The tween engine

Tweens are pooled (TweenPool) and ticked by UITick, a single central service using unscaled delta time — no per-component Update(). The runtime FPS cap in settings throttles it if needed.

Invariant

Tween lifetime. Bind tweens to their targets with tween.SetTarget(owner, get, set) so a destroyed target self-stops the tween instead of throwing every frame. UITick logs-and-drops any tickable that throws — loudly, once.

Flow graphs

Screen navigation as a state machine: nodes show views, edges fire on triggers, history enables Back.

→

Pieces

TypeRole

FlowControllerPlays a graph (a runtime clone, so mutations never touch the asset). Global or local scope, history stack with GoBack(), exposes the active node for debugging.

UINodeThe workhorse: shows/hides views on enter/exit, listens for its edges' triggers, advances when one fires.
PortalNodeGlobal listener, active for the whole flow — jumps the flow from anywhere when its trigger fires (pause menus, error popups), optionally clearing the back history.
StickyNoteNon-executable annotation in the graph.
FlowEdgeConnection: target node name, trigger, back-allowed flag, weight, optional named transition.

buttonsignaltoggleOntoggleOff viewShownviewHiddenbacktimercustom…
Extensibility seam

Register your own trigger kinds via NeoTriggerKinds.Register — implement INeoTriggerKind (Connect, Matches), optionally ITriggerKindIdDatabase to get an id picker in the editor. Flow node kinds themselves are registry-driven too (FlowNodeKinds).

What a UINode does

On enter, a UINode hides its hideViews list, shows its showViews list (with transition choreography if the edge named one), then listens for its output-edge triggers. On exit, it optionally hides what it showed (hideShownViewsOnExit) — and views shared with the next node carry over untouched, so navigating between two screens that both show the HUD never hide-and-reshow-flashes it.

Authoring in the Flow Graph Editor

Creating nodes: right-click the canvas for a search-as-you-type node picker, or drag off an output port into empty space — the picker appears and the new node is auto-wired to that port. Double-clicking a FlowGraph asset also opens the window.

Configuring edges: each edge carries its target node (by name), its trigger, an allowsBack flag (may GoBack() return across it? default yes), an optional port label, a weight (for weighted-random selection), and an optional named transition — empty means the project default.

Toolbar: Validate, Arrange (auto-layout), and Frame All / Frame Selection. In play mode the window pulses live: the active node highlights in place, traversed edges flash, and a breadcrumb strip tracks the controller's history.

Running a flow: FlowController

Add a FlowController to a scene object and assign a graph. It clones the asset at start, so runtime state never touches the asset. Key inspector options:

OptionMeaning
flowTypeGlobal (findable from anywhere via FlowController.GetGlobalController()) or Local (a self-contained sub-flow, e.g. inside one widget).
onEnableBehaviour / onDisableBehaviourWhat enabling/disabling the component does: DoNothing · StartFlow · StopFlow · PauseFlow · ResumeFlow. Default: start on enable, stop on disable.
dontDestroyOnSceneChangeKeep a global flow alive across scene loads.
goBackOnBackButton / autoCreateBackInputNavigate back when the back-button signal fires, and auto-add the hardware listener if the scene lacks one.

At runtime you get StartFlow / StopFlow / PauseFlow / ResumeFlow (pause suspends timers), GoBack() (pops the history stack, only across allows-back edges), SetActiveNodeByName() for direct jumps, and read access to activeNode, previousNode, and history.

Flow in a spec

The spec's flow section describes the same graph as data — here's a node from the game-ui showcase:

"flow": {
  "name": "GameUI", "start": "MainMenu",
  "nodes": [
    { "name": "Playing", "view": "Game/HUD",
      "next": [
        { "to": "Inventory", "on": { "button": "HUD/Inventory" } },
        { "to": "Victory",   "on": { "button": "HUD/Finish" },
          "transition": "Modal/ZoomIn" },
        { "to": "MainMenu",  "on": { "back": true } }
      ] }
  ] }

Trigger forms: {"button": "Cat/Id"} · {"signal": "Cat/Name"} · {"toggleOn"/"toggleOff": "Cat/Id"} · {"viewShown"/"viewHidden": "Cat/Id"} · {"back": true} · {"timer": 3.5} (seconds in the node).

ViewTransitionAsset makes a navigation cut a first-class, named thing: it can override the outgoing view's hide and the incoming view's show, offset their timelines, cascade children, and fly NeoSharedElement-marked widgets across the cut (shared-element transitions). Edges reference transitions by name; NeoUISettings holds the project default. Eight ship with the transition library:

Fade/CrossFade/ThroughBlack Push/SlideLeftPush/SlideRightPush/SlideUpPush/SlideDown Modal/ZoomInModal/SheetUp

Like animation presets, transition assets are discovered automatically — add your own next to the shipped ones. The transitions showcase demos the library.

Signals

Named event streams that decouple senders from receivers — the framework's connective tissue.

// Send — creates the stream on first use
Signals.Send("Audio", "Muted");

// Subscribe — parameterless or with the full Signal (sender, source, timestamp, message)
Signals.On("Audio", "Muted", () => audioSource.mute = true);
Signals.On("Game", "Over", (Signal s) => Debug.Log(s.message));
TypeRole
SignalStreamA named channel (Category/Name): send, connect/disconnect receivers.
Signal / MetaSignal<T>The message — payload-less by default; MetaSignal<T> carries a typed payload.
SignalsStatic service: get-or-create streams, Send, On/Off subscription.
SignalReceiverSerializable receiver embeddable in any component — pick a stream in the inspector, hook a UnityEvent or callback.
SignalSenderComponent / SignalReceiverComponentDrop-on components: send on enable/disable/destroy, or bind a stream to a UnityEvent — no code.
SignalLoggerLogs every signal on a stream — first stop when debugging "why didn't X react".

Widgets emit signals natively: a toggle can publish Audio/Muted, a button behavior can send on click, flow edges listen for them, particle emitters can burst on them, and effects can bind signal payloads to parameters (NeoSignalParamBinding). Prefer signals over direct UnityEvent references whenever the sender and receiver live in different prefabs.

Runtime

Back navigation

One unified sink for "go back", no matter where the press came from.

BackButton.Fire() is the single entry point, publishing on the Input/BackButton stream. Three sources feed it:

Hardware

BackButtonInput converts Escape and gamepad Cancel into Fire().

Named buttons

Any UIButton whose name matches the configured back-button name fires it automatically.

Code

Call BackButton.Fire() directly; ForceFire() bypasses the cooldown.

Flow edges with the back trigger consume the press and pop the flow history. Disable()/Enable() stack, so nested "back is temporarily forbidden" scopes compose correctly, and a cooldown (configurable in settings) prevents double-fires.

Invariant

Explicit back wiring consumes the press — a screen with a wired back edge never also triggers the global fallback, so you never double-navigate. Covered by BackNavigationTests.

Theming

Tokens × variants: name your colors once, restyle the whole app by switching a variant.

The model

A Theme asset defines named color tokens (Primary, Background, TextMuted, …) and named variants (Dark, Light, seasonal skins). Each variant supplies a value for every token — the Colors tab of the Design System window edits this matrix and includes a WCAG contrast checker. Switching the active variant via ThemeService.SetVariant() re-applies every bound graphic live.

PieceRole
ThemeColorTargetBinds a Graphic's color to a token. This is how 99% of UI gets its color.
ThemeColorRefA serialized color that is either hardcoded or a token reference — used inside styles and effects.
SelectableColorSetPer-selection-state colors (Normal/Highlighted/Pressed/Selected/Disabled), each optionally a token.
TextStyle + ThemeTextStyleTargetNamed typographic styles (font, size, spacing) applied to TMP texts — edited in the Typography tab.
ShapeStyle + ThemeShapeStyleTargetNamed surface styles (radius, border, softness) applied to NeoShapes — edited in the Shapes tab.
ThemeVariantCyclerComponent that cycles variants on interaction — instant dark-mode toggle.
ThemeBundleDefinitionA prebuilt palette you can apply wholesale (Tools → Neo UI → Setup → Apply Theme Bundle…). Drop a new SO to add your own.
Good to know

When a component drives a graphic's color at runtime (a selectable animator, a progressor), its inspector shows a ColorDriverNotice warning you that hand-set colors will be overridden. Custom drivers register their detection via NeoColorDrivers.Register.

NeoShape graphics

Resolution-independent vector UI — rounded rects, rings, glyphs — rendered by one SDF shader so everything batches on a single shared material.

Shapes

RoundedRectCirclePillRingArc CheckmarkChevronCross

Each NeoShape supports borders, solid or gradient fills (Solid · LinearGradient · RadialGradient, theme-token aware via NeoGradient), soft shadows via ElevationRecipe (depth levels 0–24), radius in pixels or percent, and texture fill with Stretch or Cover fit. Glyph shapes (checkmark/chevron/cross) render as strokes. For icon glyphs, see Icons.

Effects — two tiers

Tier 1 effects animate existing graphic properties over a normalized timeline and preserve batching:

glowPulsesheenSweepgradientCyclearcSpinner cornerMorphborderPulsehueShifttransformJuice

Tier 2 (NeoShapeVariant + ShapeEffectDefinition assets) swaps in material variants for fragment-level effects — more power, but it breaks the batch for that element (the design lint counts these). NeoEffectTrigger plays effects on interaction; NeoSignalParamBinding binds signal payloads to effect parameters.

UI particles

NeoParticleEmitter spawns pooled, canvas-native particles (each a NeoShape) in batch — coin bursts, confetti, sparkles — with composable modules:

gravitydragcolorOverLifesizeOverLifecustom via IParticleModule

Trigger bursts from signals (NeoParticleBurstOnSignal) or pointer interaction (NeoParticlePointerBurst); save configurations as NeoParticleEmitterPreset assets.

Extensibility seam

Subclass NeoShapeEffect and register with ShapeEffectRegistry for custom effects; implement IParticleModule + a ParticleModuleConfig and register with ParticleEffectRegistry for custom particle behavior. Both become available in specs and inspectors automatically.

Authoring tip

When placing effects or particles in a spec, attach them to the element that triggers them. And watch layout: a vstack force-expands child width — use a grid cell size or hstack+spacer when you need intrinsic sizing, and give images a grid cell with fit: cover so they don't stretch.

Runtime

Icons

Icons are addressed by name — "play", "arrow-right", "heart", "bell" — and rendered as font glyphs or sprites, resolved through one lookup chain.

The NeoIcon component has a single icon string field; call SetIcon("pause") to swap at runtime. Forgiving aliases canonicalize common variants (home → house, close → x, delete → trash-2), and an unknown name warns and keeps the current glyph rather than showing a blank.

Where names come from

A name is resolved through four layers, first match wins:

1 · Your spritesIconMapOverlay sprite entries
→
2 · Your glyphsIconMapOverlay codepoints
→
3 · Curated set~170 pre-baked Lucide glyphs
→
4 · Full Lucideall 1,962 names

The curated set is what's pre-baked into the icon font that Setup → Create or Repair Fonts installs (and stores on NeoUISettings.iconFont); the full Lucide table is the fallback for everything else. The Icons tab of the Design System window is the visual home: browse and pick characters from the font, configure the sprite overlay, and set fallbacks.

Custom icons: the overlay

The IconMapOverlay asset injects your own art at the front of the chain without touching the framework: sprite entries map a name to a TMP sprite (with a tint flag — tint with the label color, or render full-color as-is), glyph entries map a name to a codepoint in your own font, and aliases add forgiving alternate names for either. Because overlay entries resolve first, you can also override any built-in name with your own art.

In specs and widgets

// standalone icon element
{ "icon": { "name": "play" } }

// icon on a button
{ "button": { "id": "HUD/Play", "label": "Play", "icon": "play" } }

The featured name list (185 icons) is enumerated in Assets/docs/spec-reference.md; any of the 1,962 Lucide names and your overlay entries resolve at runtime too.

Settings & cheat menus

Describe your settings screen as a catalog — groups, items, defaults — and the framework builds the rows, persists the values, and broadcasts every change as a signal.

How it fits together

CatalogSettingsCatalog / CheatCatalog asset
→
MenuPresenterbuilds rows from the widget library
→
Row + binderwidget wired to the value
→
UserSettingsServiceget / set / persist / signal

A MenuCatalog ScriptableObject declares groups (rendered as tabs) and items — each item is a category/name id, a row kind, a label, and kind-specific fields like min/max or options. At startup, SettingsMenu / CheatMenu (both thin specializations of MenuPresenter) clone one row prefab per item from the MenuWidgetLibrary, and a MenuControlBinder on each row wires the widget to the value. You never hand-build a settings screen row by row.

Row kinds

labelbuttontoggleswitch sliderstepperdropdownrebind
Extensibility seam

Row kinds live in the NeoMenuItemKinds registry — register your own kind (say, a color picker row) and it becomes available to catalogs and specs like the built-ins.

Values: reading, writing, persisting

UserSettingsService is the single door for every menu value. A read resolves in order: a live code binding if the game registered one, then the store (PlayerPrefs by default, keys namespaced NeoUI.Settings.* — or any IUserSettingsStore you inject), then the catalog's default. Hook a setting to real game state like this:

// route "Audio/Master" through the mixer instead of PlayerPrefs alone
UserSettingsService.Bind<float>("Audio", "Master",
    getter: () => GetMixerVolume(),
    setter: v  => SetMixerVolume(v),
    persist: true);  // also write the store, so the value survives restarts

// or just react to changes anywhere
Signals.On("Settings", "Video/VSync", ApplyVSync);

Every committed change fires a signal on the Settings (or Cheat) category. Sliders can additionally emit preview signals while dragging (emitOnDrag) on a separate Settings.Preview stream, so you can live-preview volume without persisting every intermediate value — the commit happens on release.

Cheat menus

A cheat menu is the same machinery pointed at a different signal category (Cheat instead of Settings), so cheats never collide with real settings. On top of that, CheatMenu adds favourites: enable them on the catalog and testers can star their most-used cheats into a quick-access group (persisted per user via the same service). Typical cheat rows are buttons (fire an action), toggles (god mode), and steppers (add coins).

Input rebinding

A rebind row binds to a New Input System action by path (e.g. Gameplay/Jump) plus a binding index for composites. The row shows the current binding's display string ("Space", "Left Mouse"); clicking it starts an interactive "press a key…" rebind through InputRebindService, which saves overrides as JSON per action asset and restores them automatically on startup. A reset removes the override and restores the default binding.

In a spec

The top-level settings and cheats sections declare whole catalogs; place a settings/cheats element in a view to embed the generated menu:

"settings": [
  { "id": "Settings/Game",
    "groups": ["Audio", "Video", "Controls"], "start": "Audio",
    "items": [
      { "slider": { "id": "Audio/Master", "group": "Audio",
                    "label": "Master Volume",
                    "min": 0, "max": 1, "value": 0.8 } },
      { "toggle": { "id": "Video/VSync", "group": "Video",
                    "label": "VSync", "value": true } },
      { "rebind": { "id": "Controls/Jump", "group": "Controls",
                    "label": "Jump", "action": "Gameplay/Jump",
                    "bindingIndex": 0 } }
    ] }
]

Useful item fields beyond the basics: tooltip (helper text), persisted: false (the game owns the value — nothing written to the store), wholeNumbers on sliders (integer values), options on dropdowns, and step on steppers. The settings-menu showcase demos all of it.

Styling the rows

The row prefabs come from the MenuWidgetLibrary asset, installed by Setup → Create or Repair Menu Widget Library — eight row prefabs plus the category tab and panel used for group navigation. Restyle or replace those prefabs (they're ordinary Neo widgets) and every menu built from the library picks up the change. Validate confirms that control bindings resolve to catalog entries and that rebind rows name a real action.

The ID Database Manager

Since everything is addressed by Category/Name, the id registries are the project's backbone — and this window is where you keep them healthy. Eight databases ship (Views, Buttons, Toggles, Sliders, Dropdowns, Panels, Tags, signal Streams; add your own via INeoIdDatabaseProvider), browsed in three panes: database → categories → names. Three tabs:

Browse & Edit

Add, rename, and delete categories and names inline, with a substring filter and a row filter (All / Orphans / Dangling).

Search

One global substring search across every database at once — "where is anything called Pause?"

Usage / Orphans

Cross-references ids against actual use: orphans are registered but unused (⚠ badges), dangling ids are used but unregistered — with quick-fix add/delete buttons.

Renames can optionally rewrite references: with the toggle on, renaming an id also updates matching references in your spec files, after a preview dialog showing exactly which files and how many references change. Prefabs and scenes are not rewritten directly — run a sync/regenerate afterward to materialize the rename. Every inspector id picker links back here too: the quick-add button registers new ids without leaving the inspector, and the jump button opens this window focused on the exact id.

Editor performance

By design there is no animated editor chrome, no editor-tick subscriptions, no reflection scans, and exactly one settings asset. Inspectors cache their GUIStyles and SerializedObjects. If you extend the editor, follow suit.

Editor

Validation

Three buckets: hard contracts that can fail CI, design lints that advise, and an interactivity lint that catches dead UI.

Hard contracts

Theme coverage (every variant × every token) · theme bindings resolve · popup database completeness · preset database completeness · generated views registered · flow graph syntax · flow cross-references (views/buttons exist) · menu bindings resolve.

Design lint (soft)

WCAG contrast on text/surface pairs · raw font sizes where a TextStyle exists · spacing off the 4/8/12/16/24/32/48/64 scale · missing icon glyphs · tap targets under 44px · extreme sprite stretch · fixed-size elements under force-expanding layouts · Tier-2 effect batching cost · particle emitter cost.

Interactivity lint (soft)

Dead buttons (no flow trigger, signal, popup, view command, or event) · dead tabs (no container, no flow trigger).

Extensibility seam

Add project-specific rules by implementing INeoValidationRule and calling NeoValidationRules.Register, choosing which bucket (Hard / Design / Interactivity) the rule runs in.

The spec format

One JSON file describes views, widgets, theme, popups, menus, and navigation. The generator turns it into real assets; the exporter turns assets back into a spec.

Bindings — connecting spec UI to game code

The bindings action (or Generate Binding Stub in the menu) derives a contract from a spec and emits a partial-class C# stub with typed accessors for its widgets. The stub lands outside the generated root, so it survives regenerates; you implement your game logic in the other half of the partial class.

The sync protocol

Sync is the standing way to push spec changes into a project someone may have hand-edited. It exports the live project, diffs against the baseline, refuses if off-spec edits exist (unless forced), three-way merges, regenerates, and rewrites the baseline. Human edits a stale spec would have wiped are preserved.

Sync With Spec…

Merge an incoming spec with your hand edits, then regenerate.

Check For Drift

Inspect what you've changed by hand since the last generate.

Capture My Edits

Fold your hand edits into the baseline — no regeneration.

Invariant

The generated root can only be redirected through NeoWorkspace.Scoped(…), which refuses the default root — so a misbehaving script or tool can never accidentally delete your committed UI. Test runs redirect to a scratch root automatically.

A file-based JSON protocol that lets external tools — including AI agents — drive the editor: generate, validate, preview, screenshot, sync.

How to use it

Editor open: toggle Tools → Neo UI → Advanced → Agent Bridge, then write Temp/neo-request.json and read Temp/neo-result.json. The bridge refreshes the AssetDatabase before each request.

Going deeper

Extensibility seams

The framework's first principle: seams, not sealed lists. Four patterns cover everything.

1 · Keyed registries

NeoKeyedRegistry<T>: built-ins seeded lazily, replace-by-key registration, invalid entries warned and ignored. Register from any [InitializeOnLoad] or runtime bootstrap.

2 · ScriptableObject discovery

Drop an asset, it's found — no registration call at all.

3 · Interfaces & virtuals

Implement an interface or override a hook on your own component.

4 · Partials

Wizard internals (NeoSetupWizard, NeoResetWizard) are partial classes you can extend.

Promise

If you ever hit a fixed list you can't extend without editing framework code, that's considered a design bug — every set of kinds, presets, tokens, and nodes is supposed to have a seam. Report it rather than forking.

Going deeper

Testing

Test your own screens with confidence — the framework isolates test runs from your real assets and runs headless in CI.

The framework's own behavior is guarded by its internal suites — flow playthroughs that click every edge, round-trip merge safety, back-navigation rules, enable-order races. You don't need to re-test any of that; focus your tests on your screens and your game logic.

Your assets are safe

Test runs redirect all generation to a scratch root (Assets/NeoUITestScratch) and, in PlayMode, swap NeoUISettings for an in-memory clone — so your committed UI survives every test run untouched. If your own tests generate UI, use the same mechanisms (NeoTestScratchRoot in EditMode, NeoPlayModeScratchSettings in PlayMode) and you get the same guarantee.

Running headless

Unity.exe -batchmode -nographics -runTests ^
  -testPlatform EditMode -projectPath .   # or PlayMode
Tip

PlayMode tests are the right tool for "it renders but does nothing" bugs — EditMode can't catch the enable-order and input problems that only appear in a live scene.
