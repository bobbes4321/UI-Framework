Neo UI Framework

A production-grade UI framework for Unity: addressable views, signal-driven wiring, a visual flow graph for navigation, a token-based theme system, and SDF vector graphics that batch on a single material. Everything can be authored two ways — by hand in the editor, or from a JSON spec that generates (and safely re-generates) the whole UI.

Extensible by designEvery fixed set — widget kinds, presets, theme tokens, flow nodes — has a registration seam. You never fork the package to add your own.
Addressed by nameViews, buttons, signals and streams are identified by Category/Name strings, not object references or GUIDs. Wiring survives refactors and is readable in plain JSON.
Spec ⇄ prefab round-tripA JSON spec is the source of truth; prefabs are its materialization. Generate, export, diff, and three-way-merge without losing hand edits.
Loud failures, quiet editorEvery string-addressed lookup that misses logs a warning. Editor tooling never polls, never scans with reflection, never animates chrome.

What's in the box

Containers

Views, panels, popups, tooltips, overlays, and responsive roots with show/hide transitions.

Widgets

Buttons, toggles, sliders, dropdowns, tabs, steppers, counters, badges — all id-addressed and signal-emitting.

Animation

Five-channel UIAnimation (move / rotate / scale / fade / color), preset library, pooled tween engine.

Flow graphs

A node-based state machine for screen navigation with triggers, history, and named view transitions.

Signals

Named event streams (Category/Name) decoupling senders from receivers, with optional typed payloads.

Theming

Color tokens × variants (Dark/Light/…), text styles, shape styles — switch a variant and the whole UI re-skins live.

NeoShape

SDF vector shapes (rounded rects, pills, rings, glyphs) with gradients, borders, shadows, effects and UI particles — one shared material.

Editor suite

Hub, Flow Graph editor, Design System window, Gallery, Palette, Command Palette, ID Database manager, setup wizards.

Agent Bridge

A JSON request/response protocol so tools and AI agents can generate, validate, preview, and sync UI with the editor open or headless.

Getting started

From an empty project to a themed, navigable UI in five steps.

1 · Open the Hub

Tools → Neo UI → Hub is the central launcher. Every window, wizard, and advanced action is organized there by category (Author / Data / Setup / Advanced), with search. If you only remember one menu item, remember this one — or press Ctrl+K for the Command Palette, which finds views, widgets, and windows by typing.

2 · Run project setup

Tools → Neo UI → Setup → New Project Setup… walks you through choosing a theme and installing the pieces you want. It creates the single settings asset (Resources/NeoUISettings), the starter kit prefabs, widget presets, animation library, fonts, effects, and transition library. Each piece can also be created or repaired individually from the same Setup menu, and Reset To Clean Slate… uninstalls anything setup created, per component.

3 · Learn from the showcases

Every user-visible feature has a demo scene under Assets/Showcases/ — buttons, toggles, tabs, popups, animations, effects, particles, transitions, a settings menu, and a complete game UI with flow navigation. Open one, press play, then read its spec in Assets/Showcases/Specs/ to see exactly how it was described. See Showcases.

4 · Build your first screen — pick a path

Native authoring: drag widgets from the Palette window (or use starter-kit prefabs), give each an id via the inspector's category/name picker, and wire navigation in the Flow Graph Editor.

Spec authoring: write a JSON spec (there's a JSON Schema at Assets/docs/neo-spec.schema.json for IDE autocomplete) and run Tools → Neo UI → Advanced → Generate From Spec…. The generator creates prefabs, registers ids, builds flow graphs, and writes a baseline for future safe re-generation. See The spec format.

5 · Validate

Tools → Neo UI → Advanced → Validate runs hard contract checks (theme coverage, flow references, dead ids) plus soft design lints (WCAG contrast, tap targets, spacing scale) and an interactivity lint that flags buttons and tabs that do nothing. Run it after every generate. See Validation.

Tip

The framework uses the New Input System exclusively. If pointer events aren't reaching your UI, check that your EventSystem uses the InputSystemUIInputModule, not the legacy StandaloneInputModule.

Core concepts

Category/Name ids

Almost everything is addressed by a two-part string id: a category and a name — Menu/Main, Shop/BuyButton, Audio/Muted. Typed id classes exist per widget family (ViewId, ButtonId, ToggleId, SliderId, DropdownId, PanelId, TagId, StreamId), each backed by an IdDatabase asset that powers searchable pickers in every inspector, with quick-add and jump-to buttons.

Ids also drive naming: a component implementing INeoIdOwner gets a canonical GameObject name derived by NeoIdNaming (Type-Category_Name, e.g. UIButton-Shop_Buy), shared between hand authoring and generation — so hierarchies stay predictable and greppable.

Signals as glue

Components rarely hold references to each other. Instead they publish and subscribe to signal streams — named channels like Audio/Muted. A button click, a toggle flip, a flow-graph transition, a particle burst: all can be expressed as "send this signal" / "react to this signal". This keeps prefabs self-contained and makes wiring visible as data.

Spec is source of truth; prefab is materialization

When you use the spec workflow, the JSON spec describes the UI and the generator materializes it into prefabs, id registrations, presets, and flow assets. A baseline file (.neo-baseline.json) records the spec the assets were last generated from, so the framework can always tell what you edited by hand versus what the spec says — and merge instead of clobbering. Hand-authoring without specs is fully supported too; the two paths produce identical widgets (verified by round-trip tests).

WYSIWYG baking

The state you see in the prefab is the state you get at runtime start: colors are baked, start values are serialized, disabled things start disabled. There is no "wait one frame for the theme to kick in" flash.

Invariant

No silent failures. Any string-addressed lookup that misses — showing an unknown view, starting an unknown flow, sending to a misspelled command — logs a warning naming what was looked up. If something "does nothing", the Console will say why.

Invariant

Lifecycle ordering. Registries populate in OnEnable; anything that commands another object defers to Start(), because enable order across objects is arbitrary. Follow the same rule in your own scripts to avoid enable-order races.
