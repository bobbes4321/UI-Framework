Theming

Tokens × variants: name your colors once, restyle the whole app by switching a variant.

The model

A Theme asset defines named color tokens (Primary, Background, TextMuted, …) and named variants (Dark, Light, seasonal skins). Each variant supplies a value for every token — the Colors tab of the Design System window edits this matrix and includes a WCAG contrast checker. Switching the active variant via ThemeService.SetVariant() re-applies every bound graphic live.

PieceRole
ThemeColorTargetBinds a Graphic's color to a token. This is how 99% of UI gets its color.
ThemeColorRefA serialized color that is either hardcoded or a token reference — used inside styles and effects.
SelectableColorSetPer-selection-state colors (Normal/Highlighted/Pressed/Selected/Disabled), each optionally a token.
TextStyle + ThemeTextStyleTargetNamed typographic styles (font, size, spacing) applied to TMP texts — edited in the Typography tab.
ShapeStyle + ThemeShapeStyleTargetNamed surface styles (radius, border, softness) applied to NeoShapes — edited in the Shapes tab.
ThemeVariantCyclerComponent that cycles variants on interaction — instant dark-mode toggle.
ThemeBundleDefinitionA prebuilt palette you can apply wholesale (Tools → Neo UI → Setup → Apply Theme Bundle…). Drop a new SO to add your own.
Good to know

When a component drives a graphic's color at runtime (a selectable animator, a progressor), its inspector shows a ColorDriverNotice warning you that hand-set colors will be overridden. Custom drivers register their detection via NeoColorDrivers.Register.

NeoShape graphics

Resolution-independent vector UI — rounded rects, rings, glyphs — rendered by one SDF shader so everything batches on a single shared material.

Shapes

RoundedRectCirclePillRingArc CheckmarkChevronCross

Each NeoShape supports borders, solid or gradient fills (Solid · LinearGradient · RadialGradient, theme-token aware via NeoGradient), soft shadows via ElevationRecipe (depth levels 0–24), radius in pixels or percent, and texture fill with Stretch or Cover fit. Glyph shapes (checkmark/chevron/cross) render as strokes. For icon glyphs, see Icons.

Effects — two tiers

Tier 1 effects animate existing graphic properties over a normalized timeline and preserve batching:

glowPulsesheenSweepgradientCyclearcSpinner cornerMorphborderPulsehueShifttransformJuice

Tier 2 (NeoShapeVariant + ShapeEffectDefinition assets) swaps in material variants for fragment-level effects — more power, but it breaks the batch for that element (the design lint counts these). NeoEffectTrigger plays effects on interaction; NeoSignalParamBinding binds signal payloads to effect parameters.

UI particles

NeoParticleEmitter spawns pooled, canvas-native particles (each a NeoShape) in batch — coin bursts, confetti, sparkles — with composable modules:

gravitydragcolorOverLifesizeOverLifecustom via IParticleModule

Trigger bursts from signals (NeoParticleBurstOnSignal) or pointer interaction (NeoParticlePointerBurst); save configurations as NeoParticleEmitterPreset assets.

Extensibility seam

Subclass NeoShapeEffect and register with ShapeEffectRegistry for custom effects; implement IParticleModule + a ParticleModuleConfig and register with ParticleEffectRegistry for custom particle behavior. Both become available in specs and inspectors automatically.

Authoring tip

When placing effects or particles in a spec, attach them to the element that triggers them. And watch layout: a vstack force-expands child width — use a grid cell size or hstack+spacer when you need intrinsic sizing, and give images a grid cell with fit: cover so they don't stretch.

Icons

Icons are addressed by name — "play", "arrow-right", "heart", "bell" — and rendered as font glyphs or sprites, resolved through one lookup chain.

The NeoIcon component has a single icon string field; call SetIcon("pause") to swap at runtime. Forgiving aliases canonicalize common variants (home → house, close → x, delete → trash-2), and an unknown name warns and keeps the current glyph rather than showing a blank.

Where names come from

A name is resolved through four layers, first match wins:

1 · Your spritesIconMapOverlay sprite entries
→
2 · Your glyphsIconMapOverlay codepoints
→
3 · Curated set~170 pre-baked Lucide glyphs
→
4 · Full Lucideall 1,962 names

The curated set is what's pre-baked into the icon font that Setup → Create or Repair Fonts installs (and stores on NeoUISettings.iconFont); the full Lucide table is the fallback for everything else. The Icons tab of the Design System window is the visual home: browse and pick characters from the font, configure the sprite overlay, and set fallbacks.

Custom icons: the overlay

The IconMapOverlay asset injects your own art at the front of the chain without touching the framework: sprite entries map a name to a TMP sprite (with a tint flag — tint with the label color, or render full-color as-is), glyph entries map a name to a codepoint in your own font, and aliases add forgiving alternate names for either. Because overlay entries resolve first, you can also override any built-in name with your own art.

In specs and widgets

// standalone icon element
{ "icon": { "name": "play" } }

// icon on a button
{ "button": { "id": "HUD/Play", "label": "Play", "icon": "play" } }

The featured name list (185 icons) is enumerated in Assets/docs/spec-reference.md; any of the 1,962 Lucide names and your overlay entries resolve at runtime too.
