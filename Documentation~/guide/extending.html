Extensibility seams

The framework's first principle: seams, not sealed lists. Four patterns cover everything.

1 · Keyed registries

NeoKeyedRegistry<T>: built-ins seeded lazily, replace-by-key registration, invalid entries warned and ignored. Register from any [InitializeOnLoad] or runtime bootstrap.

2 · ScriptableObject discovery

Drop an asset, it's found — no registration call at all.

3 · Interfaces & virtuals

Implement an interface or override a hook on your own component.

4 · Partials

Wizard internals (NeoSetupWizard, NeoResetWizard) are partial classes you can extend.

Promise

If you ever hit a fixed list you can't extend without editing framework code, that's considered a design bug — every set of kinds, presets, tokens, and nodes is supposed to have a seam. Report it rather than forking.

Testing

Test your own screens with confidence — the framework isolates test runs from your real assets and runs headless in CI.

The framework's own behavior is guarded by its internal suites — flow playthroughs that click every edge, round-trip merge safety, back-navigation rules, enable-order races. You don't need to re-test any of that; focus your tests on your screens and your game logic.

Your assets are safe

Test runs redirect all generation to a scratch root (Assets/NeoUITestScratch) and, in PlayMode, swap NeoUISettings for an in-memory clone — so your committed UI survives every test run untouched. If your own tests generate UI, use the same mechanisms (NeoTestScratchRoot in EditMode, NeoPlayModeScratchSettings in PlayMode) and you get the same guarantee.

Running headless

Unity.exe -batchmode -nographics -runTests ^
  -testPlatform EditMode -projectPath .   # or PlayMode
Tip

PlayMode tests are the right tool for "it renders but does nothing" bugs — EditMode can't catch the enable-order and input problems that only appear in a live scene.
