Spec workflow

The spec format

One JSON file describes views, widgets, theme, popups, menus, and navigation. The generator turns it into real assets; the exporter turns assets back into a spec.

Reference docs

The always-current schema lives in the repo: Assets/docs/spec-reference.md (human-readable, generated from code) and Assets/docs/neo-spec.schema.json (JSON Schema draft-07 — point your IDE at it for autocomplete and inline validation).

Top-level structure

SectionContents
themeTheme bundle, color tokens, variants.
presetsAnimation preset definitions.
viewsThe screens: id, background token, show/hide animations, element tree.
popupsModal and popup definitions.
settings / cheatsSettings-menu and cheat-menu catalogs (sliders, toggles, dropdowns, rebind rows).
breakpointsResponsive breakpoints.
flowThe navigation state machine: nodes, edges, triggers, transitions.

Element kinds

Every element is written as { "kind": { …fields } } — the kind is the wrapper key:

vstackhstackgridoverlaypanellistspacersafearea texticonimageshape buttontoggleswitchsliderstepperdropdowninput tabtabbarprogresscounter settingscheats

(scroll is accepted as an alias for list.)

Bindings — connecting spec UI to game code

The bindings action (or Generate Binding Stub in the menu) derives a contract from a spec and emits a partial-class C# stub with typed accessors for its widgets. The stub lands outside the generated root, so it survives regenerates; you implement your game logic in the other half of the partial class.

The reason you can regenerate without fear: the baseline, the diff, and the three-way merge.

The pieces

ToolRole
SpecBaseline.neo-baseline.json in the generated root — the spec the assets were last generated from. Rewritten by every successful generate, sync, or Fold Edits.
SpecDiffNode-by-node comparison of two specs, with addressable paths like views/Menu/Main/elements[2]/background.
SpecMergeThe three-way merge: baseline + live project + incoming spec.
OffSpecLintFlags hand edits that can't be expressed in a spec and would be dropped by a merge.

The sync protocol

Sync is the standing way to push spec changes into a project someone may have hand-edited. It exports the live project, diffs against the baseline, refuses if off-spec edits exist (unless forced), three-way merges, regenerates, and rewrites the baseline. Human edits a stale spec would have wiped are preserved.

Sync With Spec…

Merge an incoming spec with your hand edits, then regenerate.

Check For Drift

Inspect what you've changed by hand since the last generate.

Capture My Edits

Fold your hand edits into the baseline — no regeneration.

Invariant

The generated root can only be redirected through NeoWorkspace.Scoped(…), which refuses the default root — so a misbehaving script or tool can never accidentally delete your committed UI. Test runs redirect to a scratch root automatically.

Spec workflow

Agent Bridge

A file-based JSON protocol that lets external tools — including AI agents — drive the editor: generate, validate, preview, screenshot, sync.

How to use it

Editor open: toggle Tools → Neo UI → Advanced → Agent Bridge, then write Temp/neo-request.json and read Temp/neo-result.json. The bridge refreshes the AssetDatabase before each request.

Headless:

Unity.exe -batchmode -projectPath . ^
  -executeMethod Neo.UI.Editor.AgentBridge.RunBatch ^
  -neoRequest req.json -neoResult res.json

{ "action": "sync", "incoming": "spec.json" }

// render without committing anything

Extensibility seam

Register custom actions (or override built-ins by id) with AgentBridgeActions.Register. Unknown actions fail loudly with the list of available ones.

Spec workflow

Showcases

Living documentation: every user-visible feature demos in a showcase scene, generated from a committed spec.

Each showcase lives under Assets/Showcases/{id}/ with its own Generated/ root and scene — isolated roots mean regenerating one never collides with another. ShowcaseDefinition ScriptableObjects are lazily discovered by ShowcaseRegistry; drop a new one to add your own.

SpecDemonstrates
buttonsButton variants, sizes, icons, badges.
toggles-switchesBoolean controls and signal wiring.
tabs-panelsTab controls and panel switching.
popupsModal confirms and rich popups.
settings-menuSettings/cheats catalogs — sliders, toggles, dropdowns, rebinding.
animations / presetsThe animation preset library in motion.
transitionsNamed view transitions between screens.
effectsShape effects (glow, sheen, spinners, morphs).
particlesUI particle emitters and modules.
game-uiA complete game UI — menus, HUD, shop, inventory — with flow navigation.
color-a-cubeA themed game-shop UI with custom tokens.
Workflow rule

Regenerate a showcase with the regenerateShowcase bridge action (not plain generate, which targets the shared default root). And preview before you commit — render the spec with the preview action and check the images across resolutions instead of trusting the JSON.
